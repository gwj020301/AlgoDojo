# 需求文档 — 算法道场 (AlgoDojo)

## 简介

算法道场（AlgoDojo）是一个面向 0 基础小白系统学习算法思路、并训练面试手撕能力的在线刷题判题平台。其核心是一个可公网部署的在线判题系统（Online Judge）：用户在网页内嵌的代码编辑器中作答，后端通过独立 Docker 容器沙箱安全执行用户代码，并用预置测试用例自动判题。

平台在判题能力之上叠加面向初学者的学习辅助体系（专题路线图、分层提示、套路速查），帮助用户从"不知道怎么想"到"能独立手撕"。

本文档仅覆盖**第一版（MVP）**范围。

### 技术栈（约束基线）
- 后端：Python / FastAPI（异步调度判题任务）
- 前端：TypeScript Web 应用（内嵌 Monaco 代码编辑器）
- 判题执行：独立 Docker 容器沙箱
- 认证：GitHub OAuth + JWT
- 作答语言：Python、TypeScript

### 术语表
- **判题（Judge）**：执行用户提交的代码并用测试用例验证正确性的过程。
- **AC（Accepted）**：所有测试用例通过。
- **专题（Topic）**：算法分类，如哈希、双指针、滑动窗口、动态规划等。
- **路线图（Roadmap）**：按难度梯度编排的学习路径。
- **分层提示（Layered Hints）**：逐级解锁的提示（思路方向 → 关键步骤 → 伪代码 → 完整题解）。

### Out of Scope（第二阶段，不在本版）
- **复习机制**：基于遗忘曲线/错题的自动复习调度（今日复习、复习时间推算等）。
- **错题本增强**：错题本的详细思路笔记/复盘记录。
- AC 代码自动归档到用户的 GitHub 仓库 + GitHub 贡献墙打卡。

## 需求列表

### 需求 1：用户认证（GitHub OAuth）

**用户故事：** 作为一名用户，我希望用 GitHub 账号登录，以便无需注册即可拥有独立的刷题数据。

#### 验收标准
1. WHEN 用户点击"使用 GitHub 登录" THEN 系统 SHALL 将用户重定向到 GitHub OAuth 授权页。
2. WHEN GitHub 授权回调成功 THEN 系统 SHALL 获取用户的 GitHub 唯一标识、用户名与头像，并创建或更新对应用户记录。
3. WHEN 用户首次登录 THEN 系统 SHALL 为该用户创建独立的刷题进度数据。
4. WHEN 认证成功 THEN 系统 SHALL 签发 JWT 并返回前端用于后续请求鉴权。
5. WHEN 请求携带无效或过期的 JWT 访问受保护接口 THEN 系统 SHALL 返回 401 未授权。
6. IF GitHub OAuth 授权失败或被用户拒绝 THEN 系统 SHALL 返回明确的错误提示并停留在登录页。
7. WHERE 第一版范围内，系统 SHALL NOT 申请对用户 GitHub 仓库的写权限。

---

### 需求 2：题库管理

**用户故事：** 作为一名用户，我希望浏览按专题分类的题库，以便有条理地选择练习题目。

#### 验收标准
1. THE 系统 SHALL 以 hot100.md（LeetCode Hot 100）为初始题库导入题目数据。
2. THE 系统 SHALL 为每道题目存储：题号、标题、题干描述、所属专题、难度、测试用例、参考题解、可用作答语言。
3. WHEN 用户访问题库列表 THEN 系统 SHALL 按专题分组展示题目及其完成状态（未开始 / 进行中 / 已通过）。
4. WHEN 用户打开某道题目 THEN 系统 SHALL 展示题干、可选作答语言与代码编辑器。
5. THE 系统 SHALL 支持按专题、难度、完成状态筛选题目。
6. IF 某题目缺少对应语言的测试用例或模板 THEN 系统 SHALL 在该语言不可选时给出提示。

---

### 需求 3：在线代码编辑与作答

**用户故事：** 作为一名用户，我希望在网页内嵌编辑器里用 Python 或 TypeScript 写代码，以便贴近真实面试手撕环境。

#### 验收标准
1. THE 系统 SHALL 在题目页提供内嵌代码编辑器（语法高亮、行号、基础编辑能力）。
2. WHEN 用户选择作答语言（Python 或 TypeScript）THEN 系统 SHALL 加载该语言对应的初始代码模板。
3. WHEN 用户切换作答语言 THEN 系统 SHALL 切换模板，并在切换前提示当前代码可能丢失。
4. WHILE 用户编辑代码 THE 系统 SHALL 在本地（浏览器）暂存草稿，防止刷新丢失。
5. THE 系统 SHALL 提供"运行（自测样例）"与"提交（全部用例判题）"两个操作。

---

### 需求 4：判题执行（沙箱）—— 核心

**用户故事：** 作为一名用户，我希望提交代码后系统能安全、快速地用测试用例判定对错，以便即时获得反馈。

#### 验收标准
1. WHEN 用户提交代码 THEN 系统 SHALL 在独立 Docker 容器沙箱中执行该代码，且容器与宿主机及其他用户任务相互隔离。
2. THE 系统 SHALL 对每次执行设置 CPU 时间/墙钟超时限制，并 WHEN 执行超过限制 THEN 终止执行并返回"超时（TLE）"。
3. THE 系统 SHALL 对每次执行设置内存上限，并 WHEN 超过内存限制 THEN 终止执行并返回"内存超限（MLE）"。
4. THE 沙箱容器 SHALL 默认禁用网络访问，并以最小权限运行（非 root、只读文件系统、受限系统调用）。
5. WHEN 代码通过全部测试用例 THEN 系统 SHALL 返回结果"通过（AC）"。
6. WHEN 代码未通过某个用例 THEN 系统 SHALL 返回"未通过"，并展示首个失败用例的输入、期望输出与实际输出。
7. IF 代码编译失败或运行时抛出异常 THEN 系统 SHALL 分别返回"编译错误"或"运行时错误"及相应错误信息。
8. WHEN 判题任务被提交 THEN 系统 SHALL 异步调度执行，并允许前端轮询或订阅判题结果状态（排队中 / 执行中 / 完成）。
9. THE 系统 SHALL 在判题完成后回收并销毁对应的沙箱容器资源。
10. IF 同一时间提交量超过判题并发上限 THEN 系统 SHALL 将任务排队而非拒绝或崩溃。

---

### 需求 5：作答记录与进度追踪

**用户故事：** 作为一名用户，我希望系统记录我每道题的作答与通过情况，以便追踪学习进度。

#### 验收标准
1. WHEN 用户提交并完成判题 THEN 系统 SHALL 为该用户保存一条提交记录（题目、语言、代码、判题结果、用时、时间戳）。
2. WHEN 用户首次 AC 某题 THEN 系统 SHALL 将该题在该用户下的状态更新为"已通过"。
3. WHEN 用户提交未通过 THEN 系统 SHALL 将该题状态置为"进行中"。
4. THE 系统 SHALL 向用户展示总体进度统计（已通过题数、按专题完成率）。
5. THE 系统 SHALL 允许用户查看某道题目的历史提交记录。

---

### 需求 6：专题路线图

**用户故事：** 作为一名 0 基础小白，我希望有一条按难度梯度编排的学习路径，以便知道先学什么、再学什么。

#### 验收标准
1. THE 系统 SHALL 提供一条按专题难度梯度编排的路线图（如 哈希 → 双指针 → 滑动窗口 → … → 动态规划）。
2. THE 系统 SHALL 为路线图上的每个专题提供配套的套路总结说明。
3. WHEN 用户查看路线图 THEN 系统 SHALL 标注每个专题的完成进度。
4. THE 系统 SHALL 为每个专题列出推荐的练习题目顺序。

---

### 需求 7：分层提示系统

**用户故事：** 作为一名用户，我希望在卡住时逐级查看提示而不是直接看答案，以便锻炼独立思考能力。

#### 验收标准
1. THE 系统 SHALL 为每道题提供分层提示，层级依次为：思路方向 → 关键步骤 → 伪代码 → 完整题解。
2. WHEN 用户请求提示 THEN 系统 SHALL 一次仅解锁下一层，且默认不直接展示完整题解。
3. WHEN 用户解锁某一层提示 THEN 系统 SHALL 记录该题的提示使用情况。
4. THE 系统 SHALL 在用户解锁"完整题解"前给出确认提示（避免误操作直接看答案）。

---

### 需求 8：套路/模板速查

**用户故事：** 作为一名用户，我希望快速查阅常见算法套路的代码模板与口诀，以便手撕时套用。

#### 验收标准
1. THE 系统 SHALL 提供一个套路速查库，收录常见套路（如滑动窗口、二分查找、回溯等）。
2. THE 系统 SHALL 为每个套路提供 Python 与 TypeScript 的可复用代码模板及要点口诀。
3. WHEN 用户在题目页查看 THEN 系统 SHALL 提供入口跳转到与该题专题相关的套路模板。
4. THE 系统 SHALL 支持按关键词检索套路模板。

---

## 非功能性需求

### 安全
1. THE 系统 SHALL 将所有用户提交的代码视为不可信，并仅在隔离沙箱中执行。
2. THE 系统 SHALL 对受保护接口进行 JWT 鉴权与用户数据隔离（用户只能访问自己的数据）。
3. THE 系统 SHALL NOT 在日志中明文记录 JWT、OAuth token 等敏感凭据。

### 性能
1. THE 系统 SHALL 在正常负载下于数秒内返回单次判题结果。
2. THE 系统 SHALL 支持判题任务并发排队，避免高并发提交导致服务不可用。

### 可部署性
1. THE 系统 SHALL 可通过容器化方式部署到公网环境。
2. THE 判题沙箱镜像 SHALL 包含 Python 与 TypeScript 运行所需的运行时。
